## Finding 1: [critical] src/orchestrator/e2e.ts:386 — e2e-author trust-boundary check trusts the agent's own self-declared manifest, letting it merge arbitrary files anywhere in the repo onto the shared staging branch

**Quote:** `const stray = changed.filter((f) => !f.startsWith(testDirPrefix) && !allowedSpecPaths.has(f));`

**Why:** Decision 39's W5 hardening exists to stop the untrusted, unreviewed e2e-author agent (Bash+Write+live-browser MCP tool access per agents/e2e-author.md, exploring a live app whose content is a classic prompt-injection surface) from landing arbitrary changes: 'reject up front ... if it touches anything outside the spec dir that isn't itself a declared spec_path' (e2e.ts:376-381 module comment). The implementation only marks a changed file 'stray' if it neither starts with `e2e/` NOR appears in `allowedSpecPaths` — and `allowedSpecPaths` (e2e.ts:384, `const allowedSpecPaths = new Set(results.manifest.map((e) => e.spec_path));`) is built from the SAME untrusted, agent-authored `results.manifest`, with no requirement that a declared `spec_path` actually resolve under `cfg.testDir`. The only upstream guard, `assertSafeSpecPath` (e2e.ts:315-322), rejects only absolute paths and `..` segments, never enforcing testDir containment. `allowedSpecPaths` is also NOT filtered to `kind === "critical"` (that filter, `results.manifest.filter((e) => e.kind === "critical")`, only happens at e2e.ts:360 for the `critical` array later passed to `proveCriticals` at e2e.ts:396). So the agent can commit e.g. `src/malicious-backdoor.ts` to its worktree branch, list it in its own manifest as `kind: "throwaway"`, and it (a) is never proven via the base/staging fail-first proof (`proveCriticals` never sees it) and (b) is never flagged stray at e2e.ts:386 because it's in `allowedSpecPaths`. The whole branch is then merged and pushed unreviewed: e2e.ts:406 `await deps.git.mergeFfOrCommit(staging, e2eBranchName(runId));` and e2e.ts:407 `await deps.git.push("origin", staging);` — landing agent-controlled content on the shared staging branch (which later rolls up to develop/main) with zero human review and zero mechanical validation. The project's own test at src/orchestrator/e2e.test.ts:632 ('allows a merge whose extra changed files are all under testDir/ or manifest-declared spec paths') proves this is the actual, intended behavior, not an oversight — using a critical `spec_path: "checkout.spec.ts"` that is not even under `e2e/` and is accepted purely because it's manifest-declared.
