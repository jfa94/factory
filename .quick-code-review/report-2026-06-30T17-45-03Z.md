## Findings by Category

### Quality

#### [important] `src/orchestrator/e2e-sweep.ts:313` — Session-mode error re-sweep erases the blocking money-path gate and ships it

- **Reviewer**: quality
- **Quote**: `if (run.mode === "workflow" && (run.e2e_sweep?.blocking_count ?? 0) > 0) {`
- **Why**: The gate-preserving carve-out (whose comment says overwriting the prior `done`+`blocking_count>0` marker to `failed` "would drop blocking_count and erase the gate") is scoped to `run.mode === "workflow"`. In session mode an `error` verdict on the crash-recovery re-sweep falls through to the general path, which writes a bare `failed` marker with no `blocking_count`. Trace: session sweep finds a blocking money-path → crashes before adjudication → resume routes to semantic re-sweep → re-sweep errors (e.g. app won't boot) → `attempts` hits `MAX_E2E_SWEEP_ATTEMPTS` → bare `failed` marker → `finalize.ts` backstop reads `blocking_count ?? 0 === 0` → no fail-closed → broken revenue path ships to develop.
- **Fix sketch**: Make the carve-out mode-agnostic — gate on `(run.e2e_sweep?.blocking_count ?? 0) > 0` regardless of `run.mode`; the bounded `attempts` increment still lets the cap path terminate without dropping `blocking_count`.

- **Reviewer**: systemic-failure
- **Failure mode**: `over-pinned-contract`
- **Scenario**: A workflow run cascade-fails tasks (blocked-environmental / capability-budget / spec-defect wedge) and reaches finalize; the returned outcome reports `cascade_dropped: []`, hiding from the operator summary which tasks were force-failed in that invocation.
- **Quote**: `cascade_dropped: next.cascade_dropped ?? [],`
- **Chain anchors**:

  - `scripts/factory-run-runner.js:673` — `all-terminal carries cascade_dropped (this-invocation drops) — surface it, never swallow.` _(comment asserts the value is surfaced; the mismatch swallows it)_

- **Why**: `next.ts` names the field `cascade_failed` on both the `work` and `finalize` envelope variants; there is no `cascade_dropped` anywhere in the engine. So `next.cascade_dropped` is always `undefined` and the `?? #### [important] `src/cli/subcommands/scaffold.ts:490-507` — Unauthenticated semantic sweeps generate CI that always requires auth

- **Reviewer**: codex-adversarial
- **Codex severity**: medium · **Confidence**: 0.91
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: The config contract allows `e2e.semantic` with no `authSetupCommand` (driving unauthenticated/signup paths), but scaffold always injects a `setup` project, makes every `semantic-*` project depend on it, and forces `storageState`. The generated `auth.setup.ts` then fails when no auth command or E2E credentials exist, so a valid unauthenticated semantic setup produces a permanently failing nightly suite before any authored signup/public specs run.
- **Fix sketch**: Only add the setup dependency + `storageState` when auth is configured (or require auth at config-parse time when `e2e.semantic` is enabled); for unauthenticated mode generate semantic projects that run without the setup project.

#### [important] `templates/ci/e2e-nightly.yml:25-29` — Committed semantic specs are not replayed with the configured seed command

- **Reviewer**: codex-adversarial
- **Codex severity**: medium · **Confidence**: 0.88
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: `e2e.seedCommand` is passed to the in-pipeline authoring sweep so authored specs can depend on seeded fixtures, but the generated nightly workflow only installs deps/browsers and runs `npx playwright test` — it never runs the seed command before replaying the committed semantic suite. Durable CI is non-reproducible: specs that passed during the sweep can fail nightly against an unseeded DB, or exercise stale/shared data.
- **Fix sketch**: Inject a seed step before `npx playwright test` when `e2e.seedCommand` is configured (same command passed to the sweep), or require semantic specs to self-seed and drop the runner-only seed knob.
