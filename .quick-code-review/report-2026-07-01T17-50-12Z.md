## Findings by Category

### Security

#### [critical] `src/orchestrator/e2e.ts:384-386` — W5 trust-boundary stray-file check is built from the full manifest, not just critical entries, so a throwaway-labeled entry can allowlist an arbitrary file

- **Reviewer**: security
- **Quote**: `const allowedSpecPaths = new Set(results.manifest.map((e) => e.spec_path));`
- **Why**: Decision 39's W5 hardening exists to stop the untrusted, unreviewed e2e-author agent (Bash+Write+live-browser MCP tool access, exploring a live app that is a classic prompt-injection surface) from landing arbitrary changes — the module comment at `e2e.ts:376-381` says to "reject up front ... if it touches anything outside the spec dir that isn't itself a declared spec_path." But `allowedSpecPaths` (line 384) is built from the _entire_ untrusted `results.manifest` — critical and throwaway entries alike — and is never filtered to `kind === "critical"` (that filter only happens later, at line 360, for a different array). The only upstream guard, `assertSafeSpecPath`, rejects only absolute paths and `..` segments, never enforcing testDir containment. So the agent can commit e.g. `src/malicious-backdoor.ts` on its branch, declare it in its own manifest as a `throwaway` `spec_path`, and it is neither proven (critical-only proof never sees it) nor flagged stray (line 386: `changed.filter((f) => !f.startsWith(testDirPrefix) && !allowedSpecPaths.has(f))`) — the whole branch then merges (`e2e.ts:406`) and pushes to staging (`e2e.ts:407`) unreviewed. The project's own test (`src/orchestrator/e2e.test.ts:632`) proves this is the actual current behavior, not an oversight.
- **Fix sketch**: Build `allowedSpecPaths` from `critical.map(e => e.spec_path)` (the set that will actually be merged), not the full manifest.
- **Also flagged by**: quality-reviewer (independently found the identical bypass at the same two lines, `e2e.ts:384` and `e2e.ts:386`, framed as a bypassable trust-boundary logic bug — see PREMISE/EVIDENCE/TRACE/CONCLUSION in raw output)

### Silent Failures

#### [important] `src/cli/subcommands/debug.ts:440` — `debug review --record` can't distinguish "e2e passed" from "e2e was silently skipped for missing config"

- **Reviewer**: silent-failure-hunter
- **Quote**: `const e2e = await runCommittedE2e({ cwd: worktree, config: deps.config.e2e });`
- **Why**: `runCommittedE2e` returns `{kind: "skipped", reason}` when `e2e.startCommand`/`baseURL` aren't configured, and `foldE2eIntoBlockers` is a no-op on that branch. `debugReviewRecord` folds the result into `confirmedBlockers` but never persists or returns `e2e.kind`/`reason` anywhere — the `DebugEnvelope` union has no field for it, so a clean pass returns the exact same envelope (`{kind: "clean", ...}`) whether the e2e suite ran and passed or was silently skipped for missing config. An operator on a repo with unconfigured/misconfigured e2e settings will see every pass reported "clean" with no way to tell the Playwright suite never ran.
- **Fix sketch**: Thread `e2e.kind` (and `reason`, when skipped) through `DebugEnvelope`'s "clean"/"findings" variants so a skip is visibly distinct from a genuine pass.

#### [minor] `scripts/factory-run-runner.js:304` — `recordResults`' idempotent-reread recovery catch is unscoped to delivery-transport errors

- **Reviewer**: silent-failure-hunter
- **Quote**: `} catch (err) {`
- **Why**: The try block wraps both the `agent()` call and `parseEnvelope()`; the catch treats _any_ exception — a genuine post-tool transport flake, but equally a bug in `agent()`'s own plumbing or an unrelated `parseEnvelope` defect — as "maybe the mutation landed" and dispatches into the same idempotent-reread recovery path. A same-key reread correctly rethrows, but a coincidentally-advanced or terminal reread on an unrelated bug gets classified "recovered" with only a log line, masking the real defect.
- **Fix sketch**: Narrow the catch to the specific transport/flake error shapes the design doc describes, or log the caught error's full detail whenever the reread differs so a masked unrelated bug still leaves a legible trace.

### Systemic

#### [critical] `src/orchestrator/e2e.ts:248` — E2E re-authoring gate is permanently defeated by any pre-manifest authoring failure; the documented repair (`rescue --reset-e2e`) does not reopen it

- **Reviewer**: systemic-failure
- **Failure mode**: `invariant-without-repair`
- **Scenario**: The e2e-author agent crashes/times out or emits an unparseable status on its first spawn (or lists an unsafe `spec_path`) → `runE2eRecord`'s early failure branch marks the phase failed with an empty manifest → operator runs `factory rescue apply --reset-e2e` expecting a clean re-author → the next `runE2eEmit` instead reports the run's e2e phase "done" (no UI surface) with zero real coverage, and the run finalizes as if e2e passed.
- **Quote**: `if (run.e2e_phase === undefined) {` _(primary anchor)_
- **Chain anchors**:
  - `src/orchestrator/e2e.ts:248` — `if (run.e2e_phase === undefined) {` _(sole re-authoring gate)_
  - `src/orchestrator/e2e.ts:552` — `async function markFailed(` _(defines `e2e_phase` on any early failure, closing the gate)_
  - `src/orchestrator/e2e.ts:622` — `if (manifest.length === 0) {` _(false-done branch hit on every re-entry after the gate is closed)_
  - `src/rescue/apply.ts:43` — `function reopenE2ePhase(phase: E2ePhase): E2ePhase {` _(documented repair path that leaves `e2e_phase` defined, not `undefined`)_
- **Why**: `runE2eEmit`'s only gate for re-spawning the e2e-author is `e2e_phase === undefined`. But `runE2eRecord`'s early failure branches (parse rejection, empty-manifest-without-no-UI-surface, unsafe spec path, unknown task_ids) call `markFailed` _before_ any real manifest exists; `markFailed` unconditionally writes a defined `e2e_phase` object with `manifest: []`. From then on, `e2e_phase` can never again be `undefined`, so the authoring branch can never re-enter — every future call falls into the `manifest.length === 0` branch, which unconditionally calls `markDone`, silently and permanently converting a retryable authoring failure into a false "nothing UI-facing, done" verdict. The documented repair, `reopenE2ePhase`, explicitly _preserves_ `manifest`/`reopen_counts`/`attempts`, so post-repair `e2e_phase` is still defined with an empty manifest and hits the same false-done path. This falsifies Decision 39's stated intent ("the phase re-enters and re-derives instead of being stuck failed forever") for precisely the case where the author never got far enough to produce a manifest. No test exercises a second `runE2eEmit` after an early `runE2eRecord` failure, nor combines it with rescue.
- **Fix sketch**: Distinguish "failed before any manifest was produced" from "failed after producing a manifest" (e.g. a dedicated status or `manifest === undefined` vs `[]`), and have `reopenE2ePhase` clear the manifest in the former case so re-authoring can actually happen.

#### [important] `src/orchestrator/finalize.ts:221` — A "completed"-terminal run with an auto-armed (not actually merged) rollup PR never gets its PRD closed or branch GC'd, and nothing ever automatically revisits it

- **Reviewer**: systemic-failure
- **Failure mode**: `stuck-state`
- **Scenario**: develop has a required status check satisfied only by GitHub's merge queue/auto-merge → rollup's immediate squash-merge is blocked by branch policy → rollup arms `--auto` and returns `merged: false, reason: "auto-armed"` → finalize marks the run "completed" and skips PRD-close/branch-GC → the queued merge eventually lands on develop with no further factory involvement ever closing the PRD issue or deleting the run's branches.
- **Quote**: `if (rollupResult.merged) {` _(primary anchor)_
- **Chain anchors**:
  - `src/git/rollup.ts:207` — `await args.ghClient.prMergeSquash(number, { subject, body: args.body, auto: true });` _(merge only armed, not landed)_
  - `src/orchestrator/finalize.ts:221` — `if (rollupResult.merged) {` _(PRD-close/branch-GC gated on landed merge only)_
  - `src/orchestrator/finalize.ts:258` — `const finalized = await deps.state.finalize(runId, terminal);` _(run flipped terminal regardless of merge outcome, closing off automatic re-check)_
- **Why**: `finalize.ts` computes `terminal` before rollup even runs and unconditionally flips run status to that terminal regardless of merge outcome; the PRD-close/branch-GC block is gated solely on `rollupResult.merged`, so "auto-armed" skips it. Once a terminal run exists, `StateManager.finalize()` throws if re-invoked at a different status, resume refuses terminal runs, and rescue/scan treats terminal runs as already-finalized — so nothing automatically revisits this run once the queued merge actually lands. The only surfacing is a log line; it's never written into the PRD, the report, or `PartialRunReport`. This extends an already-accepted D3 gap (no-merge/ci-failing/etc.) with a new path into the same hole, without adding any mechanism to close it once the async merge completes.
- **Fix sketch**: Either don't flip terminal until the armed merge is confirmed landed (poll/webhook), or add an explicit "PRD close + branch GC" step that a later `resume`/`rescue scan` pass can trigger for auto-armed-but-since-landed runs.

### Adversarial-Codex

#### [important] `src/orchestrator/e2e.ts:384-386` — Manifest-declared files outside the e2e suite can be merged unreviewed

- **Reviewer**: codex-adversarial
- **Codex severity**: high · **Confidence**: 0.9
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: `runE2eRecord` only rejects absolute paths and `..` segments before trusting the author's manifest, then the merge guard allows any changed file that either starts with `cfg.testDir/` or is listed in `allowedSpecPaths`, which comes directly from the untrusted e2e author. A critical entry such as `checkout.spec.ts` at repo root can be declared, "proven", and merged even though it is outside the committed e2e suite directory — landing unreviewed files from the author branch into staging, violating the intended W5 trust boundary.
- **Fix sketch**: Require every `critical` manifest `spec_path` to be under `cfg.testDir/`, and make the stray-file guard allow only declared critical spec paths under that directory plus narrowly-defined support files under the same directory.

#### [important] `src/orchestrator/e2e.ts:680-724` — Throwaway Playwright tooling failures can be marked done

- **Reviewer**: codex-adversarial
- **Codex severity**: high · **Confidence**: 0.86
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: The fail-closed tooling check only examines `criticalResult`; throwaway execution can return `ok: false` from a nonzero exit or reporter `errors[]` with no per-spec failed status, but the code only derives `throwawayFailed` from failed specs. In that scenario `mappable` is empty and the phase calls `markDone`, so a broken throwaway run is reported as a successful e2e phase instead of failing or suspending visibly.
- **Fix sketch**: Add the same tooling-failure check for `throwawayResult` before deriving `throwawayFailed`; fail or suspend when `throwawayResult.ok` is false and no individual failed spec can be mapped.
