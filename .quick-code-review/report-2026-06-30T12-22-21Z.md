### Security

#### [important] `src/cli/subcommands/scaffold.ts:490` — Auth-setup project inherits `trace: retain-on-failure` and leaks the test password into the nightly artifact

- **Reviewer**: security
- **Quote**: `{ name: "setup", testMatch: /auth\.setup\.ts/ },`
- **Why**: The scaffolded `setup` project (runs `auth.setup.ts`) is emitted with no `use`/`trace` override, so it inherits the base `trace: "retain-on-failure"` from `templates/playwright.config.ts:21`. The sibling `semantic-*` projects were deliberately given `trace: "off"` (`scaffold.ts:503`) precisely because a retained trace is an uploadable artifact that leaks session secrets — but the project that handles raw credentials was overlooked. `auth.setup.ts` Option B types the password (`templates/tests/e2e/auth.setup.ts:60`, `getByLabel("Password").fill(pass)`), which is recorded in the trace action log. `templates/ci/e2e-nightly.yml` runs all projects and unconditionally uploads `playwright-report/` (7-day retention); the HTML reporter bundles failed-test traces into the report, so a failed login persists the typed `E2E_PASS` to a downloadable (public on a public repo) artifact.
- **Fix sketch**: Emit the `setup` project with `use: { trace: "off" }` (mirror the `semantic-*` override), and/or document `authSetupCommand` (Option A, no in-browser credentials) as the safer default.

### Quality

#### [important] `scripts/factory-run-runner.js:618` — Workflow-mode fix-forward halt is non-durable; `resume` skips the gate and ships the blocking money-path

- **Reviewer**: quality
- **Quote**: `if (e.recommendation === "fix-forward") {`
- **Why**: PREMISE — the halt exists "so a blocking money-path never ships invisibly" and tells the operator to "review the rollup + re-run before shipping". EVIDENCE — by the time this branch runs, `runE2eRecord` has already persisted `e2e_sweep:{status:"done"}` (`e2e-sweep.ts:265`) and did _not_ set `run.status`. TRACE — the runner returns `{suspended:true}` (a script return value, not persisted state); on the operator's sanctioned re-run, `wantsSemanticSweep` returns false on the `done` marker (`next.ts:120`), the sweep is skipped, and finalize merges the rollup (`merge: shipMode === "live"`). CONCLUSION — the recovery action the halt's own message sanctions is exactly what ships the failure. The B1 fix halts only the in-process run, not a resume.
- **Fix sketch**: Make the block durable — persist a distinct sweep status (e.g. `"blocked"`) that finalize/rollup treats as a hard ship-gate while `blocking_count > 0`, or set `run.status` to suspended so re-entry cannot silently advance; clear only once the money-path re-verifies clean.
- **Also flagged by**: codex-adversarial (`scripts/factory-run-runner.js:618-630`, severity high / confidence 0.88 — "Workflow E2E blocker is only returned to the runner, not persisted in run state")

#### [important] `scripts/factory-run-runner.js:196` — `SWEEP_OUT` author-boundary schema accepts finding status `"pass"` that the no-retry record rejects

- **Reviewer**: quality
- **Quote**: `status: { type: "string", enum: ["pass", "fail", "uncertain"] },`
- **Why**: PREMISE — `SWEEP_OUT` is declared to mirror the engine's `E2eSweepResultSchema` so a malformed result is rejected and _retried_ at the author boundary (the only retryable validation; the subsequent `--results` record is no-retry). EVIDENCE — `SWEEP_OUT` allows finding `status ∈ {pass, fail, uncertain}` (line 196), but the engine's `E2eJourneyFindingSchema` is `z.enum(["fail","uncertain"])` (`e2e-sweep.ts:87`, the C3 narrowing) — `pass` is forbidden because findings are problems-only. TRACE — an author result with a `status:"pass"` finding (the agent is even told to "Record the journey result: pass/fail/uncertain" at `agents/e2e-author.md:122`) passes the retryable `SWEEP_OUT` check; `recordEnvelope` then runs `factory run e2e --results` which calls `E2eSweepResultSchema.parse` → throws → single-attempt record dies loud. CONCLUSION — the mirror is more permissive than the engine in exactly the direction that converts a recoverable mistake into an unrecoverable run death (the verdict↔blocking cross-field invariant has the same gap).
- **Fix sketch**: Tighten the `SWEEP_OUT` findings `status` enum to `["fail","uncertain"]` to match the engine (and add the verdict↔blocking refine), or derive the JS mirror from the same source as the engine so it cannot drift more-permissive than the no-retry boundary.

- **Reviewer**: codex-adversarial
- **Codex severity**: medium · **Confidence**: 0.76
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: `resolveE2eInstall` lets an explicit `quality.setupCommand` win, then infers the package manager with prefix-only `appInstall.startsWith("pnpm")` / `startsWith("yarn")`. Common valid commands — `corepack enable && pnpm install --frozen-lockfile`, `corepack pnpm install`, or a wrapper script in a pnpm repo — fall through to `npm`, so the generated workflow installs the app with pnpm/yarn but installs `@playwright/test` with `npm i -D`.
