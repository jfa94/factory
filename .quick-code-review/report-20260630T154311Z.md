### Quality

#### [important] `src/cli/subcommands/scaffold.ts:592` — e2e.blocking deadlocks every task→staging merge

- **Reviewer**: quality
- **Quote**: `if (blocking) await ensureRequiredCheck(opts, E2E_CHECK_CONTEXT);`
- **Why**: with `e2e.blocking=true`, scaffold persists `"E2E"` into the GLOBAL `git.requiredStatusChecks` overlay. Run create reuses that same list to protect the per-run staging branch (`run.ts:395 requiredChecks: stagingDeps.config.git.requiredStatusChecks`, `strict:true`). The scaffolded `e2e.yml` only runs on develop PRs (`e2e.yml:4 branches: [develop]`), so it never reports `"E2E"` on a task PR based on `staging-<run-id>`. `shipTask` squash-merges via `gh pr merge <n> --squash` (no `--admin`); GitHub blocks it because `"E2E"` is permanently "Expected — waiting for status". Every task ship in an `e2e.blocking` repo wedges; no run reaches finalize. Default config (`requiredStatusChecks=[]`) is unaffected — regression is specific to opting into `e2e.blocking`.
- **Fix sketch**: keep `"E2E"` out of the per-run staging `requiredChecks` — develop-only check list, subtract at run-create, or trigger `e2e.yml` on `staging-*` bases too.
- **Chain anchors**:
  - `src/cli/subcommands/scaffold.ts:592` — `if (blocking) await ensureRequiredCheck(opts, E2E_CHECK_CONTEXT);` _(writes E2E into global overlay)_

### Silent Failures

#### [minor] `scripts/factory-run-runner.js:527` — Non-blocking sweep findings discarded on an `error` verdict

- **Reviewer**: silent-failure
- **Quote**: `if (sweep.verdict === "error") return { verdict: "error", journeys_total: 0, findings: [] };`
- **Why**: an e2e-author can surface real non-blocking problems (uncertain/fail journeys) then hit a mid-sweep MCP/boot failure → `error`. `normalizeSweep` collapses that to a clean empty error, zeroing `journeys_total` and dropping every finding, so the genuine problems already observed vanish from the recorded sweep. At the attempt cap the run best-effort advances to completed with an empty money-path summary; findings re-surface only if a later clean re-sweep happens to re-find them.
- **Fix sketch**: preserve already-observed non-blocking findings on the `error` collapse (don't zero `findings`/`journeys_total`).

#### [minor] `src/cli/subcommands/scaffold.ts:586` — Opted-in e2e.blocking gate silently defeated if e2e.yml indent drifts

- **Reviewer**: silent-failure
- **Quote**: `return blocking ? withEnv.replace(/^        continue-on-error: true.*\n/m, "") : withEnv;`
- **Why**: when `e2e.blocking` is true the scaffolder strips `continue-on-error: true` so a crashing app reds the job. `String.replace` returns the input unchanged on no match — no throw, no post-condition. The regex hard-codes the template's current 8-space indent; any future reindent of `templates/ci/e2e.yml` makes the strip a silent no-op, leaving `continue-on-error` in place (job green on failure), so the required `"E2E"` check passes anyway. An operator who opted into blocking gets advisory-only behaviour with zero signal.
- **Fix sketch**: assert the strip changed the text (or parse the YAML and delete the key), failing loud if `continue-on-error` survives in blocking mode.

### Systemic

#### [important] `src/orchestrator/e2e-sweep.ts:321` — Shared sweep-attempt budget lets a transient error sweep defeat the workflow money-path fix-and-reverify path

- **Reviewer**: systemic-failure
- **Failure mode**: `invariant-without-repair`
- **Scenario**: workflow run, `e2e.semantic` on, all tasks done. Sweep #1 returns `error` (app momentarily fails to boot) → attempts=1, suspended; human fixes boot + resumes; sweep #2 finds a real blocking money-path → done, `blocking_count>0`, attempts=2, suspended/fix-forward; human fixes the money-path + resumes, but `wantsSemanticSweep` sees `attempts(2) < 2 == false` so it never re-sweeps, and finalize's backstop sees `attempts(2) >= 2` → run finalized `failed` WITHOUT re-verifying the fix.
- **Quote**: `const attempts = (run.e2e_sweep?.attempts ?? 0) + 1;`
- **Chain anchors**:
  - `src/orchestrator/e2e-sweep.ts:321` — `const attempts = (run.e2e_sweep?.attempts ?? 0) + 1;` _(error path increments shared counter)_
  - `src/orchestrator/e2e-sweep.ts:280` — `attempts: (s.e2e_sweep?.attempts ?? 0) + 1,` _(blocking-done re-verify increments SAME counter)_
  - `src/orchestrator/next.ts:131` — `(run.e2e_sweep.attempts ?? 0) < MAX_E2E_SWEEP_ATTEMPTS` _(re-sweep gate reads shared counter)_
  - `src/orchestrator/finalize.ts:189` — `if ((run.e2e_sweep?.attempts ?? 0) >= MAX_E2E_SWEEP_ATTEMPTS) {` _(cap forces terminal `failed` without re-verify)_
- **Why**: `MAX_E2E_SWEEP_ATTEMPTS (=2)` is one counter incremented by both the agent-`error` retry path (line 321) and the blocking re-verify path (line 280). The carve-out at line 308 handles only the error-AFTER-blocking ordering; error-BEFORE-blocking is unguarded, so a transient error silently consumes the blocking-reverify budget and a real blocking finding gets zero re-verify cycles. Fail-safe (no bug ships), but the human's valid fix is never tested and the only recovery is a fresh full run.
- **Fix sketch**: separate the two budgets — a dedicated counter, or reset `attempts` when the marker transitions from a `failed` error record to a `done` blocking record.
