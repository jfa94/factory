## Findings by Category

### Quality

#### [minor] `scripts/factory-run-runner.js:176` — SWEEP_OUT author gate omits the verdict↔blocking cross-field check the engine enforces

- **Reviewer**: quality
- **Quote**: `verdict: { type: "string", enum: ["approve", "blocked", "error"] },`
- **Why**: PREMISE — the `SWEEP_OUT` JSON-schema gate is documented (comment at `factory-run-runner.js:197`, `This retryable author gate must never` be more permissive than the no-retry `--results` boundary) to mirror the engine's `E2eSweepResultSchema`. EVIDENCE — the gate mirrors only the `status` enum (line 200) and structure; it has **no** cross-field rule tying `verdict` to the findings' `blocking` flags, which the engine's `superRefine` (`e2e-sweep.ts:105-121`) does enforce. TRACE — an author emits `{verdict:"approve", findings:[{…blocking:true}]}` (an LLM self-contradiction); `SWEEP_OUT` accepts it → `recordEnvelope` writes it and runs `factory run e2e --results <path>` → the engine `superRefine` throws → `parseEnvelope` finds no valid envelope and throws → the **no-retry** `--results` boundary halts the workflow run ungracefully instead of re-prompting the author. CONCLUSION — a full opus+app-boot sweep is wasted and the run halts on a contradiction the retryable gate was supposed to catch.
- **Fix sketch**: After the author returns in `runE2e()`, add a small JS cross-field guard mirroring the `superRefine` (reject approve+blocking and blocked+no-blocking, treat like a schema miss so `agent` re-prompts) **before** `recordEnvelope` writes the file.

### Systemic

#### [critical] `src/orchestrator/next.ts:126` — Workflow run with a non-bootable app wedges forever: the boot-failure attempt cap is unreachable on the blocking path

- **Reviewer**: systemic-failure
- **Failure mode**: `stuck-state`
- **Scenario**: A `--workflow` run opts into `e2e.semantic` but the staging app's `/` needs a seeded DB/service container (the guide documents this as unsupported autonomously); `e2e-author` reports the boot failure as verdict `"blocked"`, `runE2eRecord` records `e2e_sweep` `done` + `blocking_count=1` and suspends, and every subsequent `factory resume` re-routes to a fresh re-sweep that boot-fails the same way — `MAX_E2E_SWEEP_ATTEMPTS` never engages and the run never reaches `completed`/`failed`.
- **Quote**: `if (run.mode === "workflow" && (run.e2e_sweep.blocking_count ?? 0) > 0) {`
- **Chain anchors**:
  - `src/orchestrator/next.ts:126` — `if (run.mode === "workflow" && (run.e2e_sweep.blocking_count ?? 0) > 0) {` _(stuck-state: the `status==="done"` branch returns at line 127, **before** the cap check at line 131, so the loop is unbounded)_
  - `src/orchestrator/next.ts:131` — `if ((run.e2e_sweep?.attempts ?? 0) >= MAX_E2E_SWEEP_ATTEMPTS) return false; // cap: best-effort` _(missing-repair: the bounding cap, unreachable whenever the marker status is `done`)_
  - `src/orchestrator/e2e-sweep.ts:239` — `if (results.verdict !== "error") {` _(trigger: a boot failure reported as `blocked` enters this no-cap branch, increments `attempts`, sets `status:"done"` + suspends, instead of the capped `error` branch below)_
  - `src/orchestrator/e2e-sweep.ts:65` — `export const MAX_E2E_SWEEP_ATTEMPTS = 2;` _(invariant: the cap whose comment claims it bounds the "app never boots" loop — defeated for the workflow blocking path)_
- **Why**: The cap at `next.ts:131` only runs when `e2e_sweep.status !== "done"`. A `blocked` sweep always records `status:"done"` (`e2e-sweep.ts:268`) while still incrementing `attempts` (line 269), so the cap is structurally unreachable on the blocking path even as `attempts` grows without bound. The only exits are a re-sweep that comes back clean (`blocking_count=0`) or one that comes back `error` — neither of which a deterministically-non-bootable app will produce. The comment at `e2e-sweep.ts:60-62` explicitly claims the cap prevents "an infinite suspend-loop when the agent repeatedly fails (e.g. the app never boots)", but that guarantee holds only for the `error` verdict, not for a boot failure surfaced as a blocking `blocked`.
- **Fix sketch**: Apply the `MAX_E2E_SWEEP_ATTEMPTS` cap inside the `status==="done"` workflow-blocking branch too (e.g. check `attempts >= cap` before re-routing at `next.ts:126-128`, falling through to finalize/fail as best-effort) so a repeatedly-non-bootable workflow run terminates.

#### [important] `src/orchestrator/e2e-sweep.ts:93-122` — `error` sweep results can hide blocking findings and advance

- **Reviewer**: codex-adversarial
- **Codex severity**: high · **Confidence**: 0.86
- **Quote**: _n/a — existence-checked (review schema has no verbatim field)_
- **Why**: `E2eSweepResultSchema` rejects contradictory `approve`/`blocked` results, but it allows `verdict: "error"` with non-empty or blocking `findings`. `runE2eRecord` then treats every `error` result as an incomplete sweep, ignores the findings entirely, and after the retry cap returns `recommendation: "advance"`, `blocking_count: 0`, `findings: []`. A realistic e2e-author output for a boot/login/checkout failure could carry the blocking finding but use `error`; this path drops that evidence and can let the run finalize as best-effort clean.
- **Fix sketch**: Make `error` mutually exclusive with findings, or treat any `error` result containing blocking findings as a recorded `blocked` sweep; add tests covering `verdict:"error"` with blocking findings through `runE2eRecord`, including the retry-cap path.
